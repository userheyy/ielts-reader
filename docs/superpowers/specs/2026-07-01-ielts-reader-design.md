# 雅思阅读精读器 — 设计文档

**日期**: 2026-07-01
**目标**: 为两个月后的雅思 6.5 备考,搭建一个本地个人英语学习网站的**第一个子系统:阅读精读器**。

---

## 1. 背景与范围

### 用户目标
- 两个月后考雅思,目标 6.5。
- 想要一个"左边英文原文、右边实时查看语法/生词/翻译"的精读体验(参考截图:左原文,右侧逐句语法拆解+翻译)。
- 读文章时遇到生词能"入库",供以后背单词。

### 本次范围(仅第一个子系统)
整个学习平台包含多个独立子系统,本次**只做阅读精读器 + 生词入库**:

- ✅ 阅读器:左原文 / 右语法拆解+翻译,点句联动、点词查释义。
- ✅ 生词库:读文章时把生词入库,可查看/搜索/删除/导出/导入。
- ✅ 预处理:由 Claude 作为"老师"读 docx,逐句生成语法+翻译+释义数据文件。

### 明确不在本次范围(后续各自独立子项目)
- ❌ 试题库 / 刷题 / 对答案(从扫描 PDF 提取结构化题目,不确定性最大,单独做)。
- ❌ 背单词 / 复习记忆算法(间隔重复 SRS),单独做——但本次数据结构**预留字段**。
- ❌ 听力、写作、口语。
- ❌ 文章配图 / 听力地图题图片(阅读器只做阅读文章文本精读)。

---

## 2. 关键前提与已验证事实

- **数据源**: 用户已有剑桥雅思 1-19 高清 PDF(含可提取文本层)。**不使用 OCR**,也**不用 docx**(见下)。
- **docx 路线已否决**: 用户手动转的 `【2】剑桥雅思真题2.docx` 中,单栏内容(引言等)转换正常,但**阅读文章正文是报纸式多栏排版,转 Word 时被横向逐行拼接,句子撕碎、语序错乱**(Passage 1/2 均验证为乱序)。故 docx 不可用。
- **PDF 文本层自动分栏亦已否决**: 尝试用 pdfplumber 按 x 坐标分栏、按竖白缝 crop 重排——因栏边界不均匀、紧密排版下相邻栏字符仍横向串接,输出不可靠。放弃一切"从文本层自动重排多栏"的做法。
- **最终采用:PDF 高清渲染 → 分栏截图 → Claude 读图抄录**。用 pypdfium2 将目标页以 scale≈4 渲染为高清 PNG,按栏裁成竖条图,Claude 直接"看图"按正确阅读顺序读出并抄录原文。已在剑2 P1 验证:该法能清晰读字、顺序正确、语义连贯。栏布局再乱也不受影响(靠人眼顺序而非坐标)。
- **公开文本资源不可用**: 经调查,GitHub 上的剑雅文本仓库(如 kbtxwer/IELTS)已于 2025-12 被剑桥大学出版社 DMCA 下架;其余来源为网页爬取(版权灰色、不稳定)或 AI 生成仿真题(非真题)。故**不依赖外部文本源**,以用户本地 PDF 为准。
- **不接任何 API**: 语法拆解/翻译/释义**不走大模型或第三方接口**。改为 Claude 在**预处理阶段**读 PDF 页面图生成数据文件,阅读器网页只展示现成数据。零联网、零费用。
- **运行时**: 本机有 Node v24、Python 3.13。预处理脚本(渲染 PDF 为图)用 Python + pypdfium2。阅读器本身是纯静态网页,运行不依赖它们。

### 本方案的代价(用户已知情)
每篇文章的原文**由 Claude 读页面图抄录**,语法/翻译数据由 Claude **逐句生成**,均非程序自动产出。剑2 一个 test 有 3 篇阅读,每篇约 900 词、十几到几十句。产出速度受此限制——这是"质量高但需逐篇产出"的模式,而非"接 API 无限自动生成"。所谓"人工校对"由 Claude 承担(读得懂英文语义、能发现串栏错误);用户只需最终扫一眼确认体验。本次先用剑2 Test 1 的 3 篇把系统跑通,不要求 19 本齐全。

## 3. 整体架构

纯本地、纯静态网页。无服务器、不联网、不接 API。浏览器直接打开 HTML 即用。

三个角色:

```
┌─────────────────────────────────────────────────────────┐
│  角色1:预处理(Claude 手工做,一次性)                     │
│  PDF 目标页高清渲染为图 → Claude 读图抄录正确原文         │
│  → 逐句写语法拆解+翻译+释义 → 输出 JSON(每篇一个)+更新索引│
└─────────────────────────────────────────────────────────┘
                          ↓  生成到 data/passages/
┌─────────────────────────────────────────────────────────┐
│  角色2:阅读器网页(静态 HTML/CSS/原生 JS)                 │
│  读 JSON → 左原文 / 右语法翻译 → 点句联动 → 点词入库      │
└─────────────────────────────────────────────────────────┘
                          ↓  生词写入
┌─────────────────────────────────────────────────────────┐
│  角色3:生词库(浏览器 localStorage)                       │
│  记录 单词/释义/例句/出处/时间 → 独立页面查看·管理·导入导出│
└─────────────────────────────────────────────────────────┘
```

### 技术选型
- **HTML + CSS + 原生 JavaScript**。不用框架、不用打包工具、无需 `npm install`。
- 数据 = 本地 JSON 文件(文章)+ 浏览器 localStorage(生词)。
- 预处理脚本用 Python(提取 docx 文本);讲解内容由 Claude 生成。

### 为什么这么选
用户是本地个人用、要马上能跑、零维护。上框架会增加构建步骤和使用门槛。纯静态文件双击即用,后续加功能也简单。

## 4. 数据文件结构(系统核心)

### 4.1 单篇文章 JSON
路径 `data/passages/<id>.json`,例 `c2-test1-p1.json`:

```json
{
  "id": "c2-test1-p1",
  "source": "剑桥雅思2 · Test 1 · Passage 1",
  "title": "Right and left-handedness in humans",
  "sentences": [
    {
      "id": 1,
      "para": 1,
      "en": "Why are some people right-handed and others left-handed?",
      "zh": "为什么有些人惯用右手,有些人惯用左手?",
      "grammar": {
        "type": "特殊疑问句(并列)",
        "note": "由 and 连接两个并列问句,后半句 others left-handed 承前省略了 are。"
      },
      "words": [
        { "w": "right-handed", "pos": "adj.", "def": "惯用右手的" },
        { "w": "left-handed", "pos": "adj.", "def": "惯用左手的" }
      ]
    }
  ]
}
```

字段说明:

| 字段 | 作用 |
|------|------|
| `id` | 文章唯一标识,用于 URL 和文件名 |
| `source` | 出处(剑X·TestY·PassageZ),显示与生词溯源用 |
| `title` | 文章标题 |
| `sentences[]` | 文章按句拆分,数组顺序即原文顺序 |
| `.id` | 句子在本文内的序号(从 1 起) |
| `.para` | 句子所属段落编号(从 1 起),用于左侧正确分段 |
| `.en` | 英文原句(左边显示) |
| `.zh` | 中文翻译(右边显示) |
| `.grammar.type` | 句子结构类型标签(主句/并列句/定语从句/插入语…) |
| `.grammar.note` | 语法拆解讲解(右边显示) |
| `.words[]` | 本句重点难词:`w` 词形 / `pos` 词性 / `def` 释义。点词与入库默认释义用 |

### 4.2 文章索引 `data/index.json`
阅读器首页读它展示"文章库"列表。每生成一篇追加一条:

```json
{
  "passages": [
    {
      "id": "c2-test1-p1",
      "source": "剑桥雅思2 · Test 1 · Passage 1",
      "title": "Right and left-handedness in humans",
      "sentence_count": 42
    }
  ]
}
```

### 设计决定
- **按句存储**: 天然支持左句↔右讲解一一对应、点击高亮联动。
- **段落归属**: 用 `para` 编号保证左侧正确分段,不连成一坨。
- **图片不搬入**: 本次不处理配图/地图题图,用户需要看图直接看 PDF。

## 5. 阅读器界面与交互

两个页面 + 一个生词库页面(共三页)。

### 5.1 首页 / 文章库 `index.html`
- 读 `data/index.json`,列出文章卡片:标题 + 出处 + 句数。
- 点卡片 → 进入 `reader.html?id=<id>`。
- 顶部有入口链接到生词库 `vocab.html`。

### 5.2 阅读器 `reader.html?id=<id>`

左右分栏布局:

```
┌──────────────────────────┬──────────────────────────┐
│  左:原文(可滚动)         │  右:讲解(按句顺序全展开)  │
│  Why are some people      │  【句1】特殊疑问句(并列)   │
│  right-handed and others  │  拆解:由 and 连接两个并列… │
│  left-handed?  ←点击高亮   │  翻译:为什么有些人惯用右手…│
│  Practically speaking,... │  生词:right-handed 惯用…  │
│  (按 para 正确分段)        │  【句2】…                  │
└──────────────────────────┴──────────────────────────┘
```

核心交互:
1. **点句子** → 该句左侧高亮;右侧对应讲解卡片滚动定位 + 高亮。主操作(逐句精读)。右侧默认全部按顺序展开,便于上下对照(非只显单句)。
2. **点单词** → 弹出小气泡:显示释义(来自该句 `words[]`;未收录则显示"未收录")+ `+ 入库` 按钮。
3. **点 `+ 入库`** → 存进生词库,自动带上:所在句 en+zh、文章出处、释义、时间。提示"已入库"。已入库的词在原文中标色(绿色下划线)避免重复。
4. **翻译显隐开关**(顶部按钮): 一键隐藏/显示右侧中文翻译,支持自测。

布局细节:
- 左右分栏,右侧讲解全展开、点左句滚动定位+高亮。
- 响应式:窗口窄时上下堆叠;以桌面/iPad 横排为主。

### 5.3 生词库 `vocab.html`
- **列表**: 表格显示所有生词(单词/释义/出处/时间),按时间倒序。
- **搜索/筛选**: 按单词或出处过滤。
- **删除**: 删误存或已掌握的词。
- **点出处** → 跳回原文那句 `reader.html?id=<id>&sentence=<n>` 复习语境。
- **导出**: 全部生词导出为 JSON 文件(备份/换浏览器/喂未来背单词功能)。
- **导入**: 从 JSON 文件恢复(合并,按 word 去重)。
- **计数**: 顶部显示"共 N 个生词"。

## 6. 生词库数据结构(localStorage)

键名 `ielts_vocab`,值为生词数组。单个生词:

```json
{
  "word": "practically",
  "def": "实际上;几乎",
  "pos": "adv.",
  "sentence_en": "Practically speaking, this has never been a problem.",
  "sentence_zh": "实际上,这从来都不是个问题。",
  "source": "剑2 · Test 1 · Passage 1",
  "passage_id": "c2-test1-p1",
  "sentence_id": 5,
  "added_at": "2026-07-01",
  "status": "new",
  "review": { "level": 0, "next_due": null, "history": [] }
}
```

**预留字段(本次不使用,为背单词子项目预留,避免将来迁移)**:
- `status`: 学习状态(new/learning/known)。
- `review`: 间隔重复所需(当前等级、下次到期、历史)。

去重键: `word`(小写)。同词重复入库时更新出处/例句,不新增。

## 7. 预处理工作流(Claude 逐篇产出)

1. **定位页码**: 用 pdfplumber 在目标 PDF 中搜文章标题(如 `AIRPORTS ON WATER`),得到 Passage 所在页(可能跨页)。
2. **高清渲染**: `tools/render_page.py` 用 pypdfium2 将该页以 scale≈4 渲染为 PNG;若为多栏,按栏裁成竖条图(每图宽度覆盖 1~2 栏),存入 scratchpad。
3. **读图抄录**: Claude 用 Read 工具读这些 PNG,**按正确阅读顺序**(栏内从上到下、栏间从左到右)抄录出英文原文,得到干净、连贯的 Passage 文本。跨页/跨栏在此人工接续。
4. **逐句产出讲解**: Claude 作为老师,对抄录好的原文**逐句**产出:英文原句 → 中文翻译 → 语法结构类型 + 拆解讲解 → 重点难词释义。
5. **写数据文件**: 写成 `data/passages/<id>.json`,并往 `data/index.json` 追加索引条目。
6. 本次先产出**剑2 Test 1 的 3 篇阅读**验证体验(数量按实际逐篇推进)。

> 注:此流程无法做成"一键全自动脚本"——读图抄录和逐句讲解是 Claude 在会话中完成的模型工作,`render_page.py` 只负责把 PDF 页变成图。

│   ├─ index.json        # 文章索引

│       └─ c2-test1-p1.json ...

├─ start.bat             # 一键启动本地静态服务器(双击)

约定:
- 项目放 `ielts-app\`,与 PDF 素材分开。
- 界面语言:按钮/标签用中文;文章内容英文;讲解中英对照。
- 因浏览器对 `file://` 下 `fetch` 本地 JSON 有跨域限制,**采用本地静态服务器方式访问**:提供一键启动脚本 `start.bat`(双击运行 `python -m http.server`),浏览器访问 `http://localhost:<port>`。数据(`data/*.json`)与代码分离,保持干净。不采用"数据内联为 JS 变量"方案。

## 9. 测试策略

- **数据校验**: 一个小脚本/页面检查每个 passage JSON 结构完整(必填字段齐、句子 id 连续、para 单调不减)。
- **交互手测清单**: 点句高亮联动、点词气泡与释义、入库后标色与去重、翻译显隐、生词库增删查、导入导出往返一致。
- **跨浏览器**: 主用浏览器(Chrome/Edge)打开三页均正常。
- 纯静态、无后端,不引入自动化测试框架;以数据校验脚本 + 手测清单为准。
